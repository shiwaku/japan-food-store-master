# 食品店POIの網羅性検証

### 〜 Overture Maps のコンビニは実数の2.2倍だった、名寄せ格闘記 〜

> **esaカテゴリ案**: `GIS/データ品質/食品店POI検証`
> **タグ案**: `Overture Maps` `OpenStreetMap` `DuckDB` `POI` `名寄せ` `データ品質` `PMTiles`
> **想定読者**: オープンPOIデータ（Overture / OSM）を実務で使おうとしているエンジニア・データ屋

## TL;DR

- 「住所を入れるだけの移動しやすさ診断」アプリの目的地レイヤーに使う**食料品店POI**を、**Overture Maps** と **OpenStreetMap** で比較検証した。
- Overtureのコンビニは **121,057件＝実数（約5.6万店）の約2.2倍** に水増しされていた。原因は「同じ実店舗が提供元ごとに別レコードとして残る」クロス提供元の重複。
- この重複は **単純な空間名寄せでは消せない**。汎用名レコードの座標が実店舗から**数百m〜500m超ズレて**おり、それは都市部の実店舗間隔より大きいため、「重複除去」と「別店舗保持」を両立できない（原理的上限）。
- 次善策として **チェーンごとに最網羅の単一提供元だけを採用**する名寄せを実装。**171,180 → 109,602件** に削減し、コンビニ54,987件は実数約56,000件とよく整合した。
- カテゴリ別に見ると **スーパーはOSM有利、コンビニ・ドラッグストアはOverture有利** と優劣が入れ替わる。単一ソースでの一元化は一長一短。
- 全処理はDuckDBで再現可能（末尾にスクリプト）。副産物として **DuckDB v1.3.2 の `ST_Distance_Spheroid` が `-nan` を返すバグ** も踏んだ。

### 🗺️ 実物を触る

- **インタラクティブ比較ビューワー（MapLibre + PMTiles）**: <https://shiwaku.github.io/japan-mobility-ease-diagnosis/compare.html>
  - カテゴリ選択＋ソース別ON/OFFで、Overture（名寄せ済）とOSMの分布・重なりを地図上で見比べられる。クリックで店名表示。

## 1. 背景：なぜ食品店POIの品質にこだわるのか

作っているのは「住所を入れるだけの移動しやすさ診断」というWebアプリ（公共交通オープンデータチャレンジ2026 応募作品）。住所を入れると、その場所の最寄りバス停・駅までの徒歩時間、便数、そして**病院・スーパーへの到達可能性**を診断カルテ形式で出す。

この「スーパーへ行けるか」を判定するには、まず**食料品店がどこにあるか**の正確なPOIが要る。全国の食料品アクセス困難人口は904万人（65歳以上の4人に1人）と推計されており、目的地データの品質は診断の土台になる。

ところが、いざオープンPOIを触ってみると、件数の水増し・重複・カテゴリのズレが山ほど出てきた。本記事はその**データ品質検証の記録**である。

### 定義を農水省に合わせる

比較の前に「食料品店とは何か」を揃える必要がある。本プロジェクトは農林水産省「食料品アクセス」の店舗定義に準拠した。

| 含む | 含まない |
|---|---|
| 食品スーパー・各種食料品小売業 | **調剤薬局**（食品を扱わない薬局） |
| コンビニエンスストア（地方では主要な食料調達先） | 100均・ディスカウント・雑貨 |
| 生鮮食料品専門店（青果・精肉・鮮魚） | 百貨店に紛れる非食品テナント |
| **ドラッグストア**（2020年推計から追加。食品を扱う） | |

ポイントは **ドラッグストア（食品扱う）は含み、調剤薬局は含まない**こと。OSMでもOvertureでも、この2つは別カテゴリ／別タグとして扱う。

## 2. データソースと取得条件

| ソース | 取得方法 | ライセンス | 生件数 |
|---|---|---|---|
| **Overture Places** | S3のGeoParquetをDuckDBで日本bbox抽出（release 2026-06-17.0） | CDLA-Permissive-2.0（再配布可） | 246,400 |
| **OpenStreetMap** | Overpass API、日本全域（`ISO3166-1=JP`） | ODbL（出典表記のうえ再配布可） | 111,186 |

補足すると、当初 **Yahoo!ローカルサーチAPI（YOLP）** も検討したが、2022年12月施行の規約でデータの保存・キャッシュが禁止されており、タイル化・再配布する本用途では使えなかった。GeoParquetで一括取得でき再配布可能なOvertureを主データにした所以である。

### 罠その1：bboxだけだと国外POIが12,000件混入する

Overtureは全世界データで、日本のbbox（経度122.5–154 / 緯度24–45.8）は**韓国・北朝鮮・ロシア沿海州・中国沿岸を含む**。素直にbboxで切ると国外POIが混ざる。

```
国別内訳（食品系全カテゴリ）: JP 234,077 / KR 12,109 / RU 160 / CN 53 / KP 1
```

`addresses[].country = 'JP'` で絞れば除去できる（この属性は欠損なし）。OSMは `ISO3166-1=JP` で最初から日本限定なので混入なし。

### 罠その2：カテゴリに非食料品が紛れる

Overtureの `discount_store` は実体がほぼ100円ショップ・雑貨（ダイソー・キャンドゥ・セリア等）。`department_store` は本物の百貨店（デパ地下＝食料品）とパルコ・ルミネ・無印・ヤマダ電機・JTB等の非食品が混在。どちらも**カテゴリごと除外**し、さらに `grocery` へ漏れ込んだ100均・証明写真機「Photo ME」を店名で除去した。

定義準拠後の件数は **Overture 170,899件 / OSM 81,363件**。ここまでで前処理は終わり……と思っていた。

## 3. 網羅性比較：件数が多い＝網羅的、ではない

「件数はOvertureが倍以上あるから網羅的だろう」——最初はそう思った。カテゴリを無視して**100m以内の空間オーバーラップ**を取ると、

| 指標 | 件数 | 割合 |
|---|---|---|
| OSM総数 | 81,363 | — |
| Overture総数 | 170,899 | — |
| OSMのうちOvertureに一致あり | 68,835 | **84.6%** |

OSMの店の84.6%はOverture側にも存在する。では残り15%は？　**Overtureが取りこぼしているOSMの店**を地域別に見ると：

| エリア | OSM総数 | Overtureに無い | 割合 |
|---|---|---|---|
| 三大都市圏 | 33,040 | 3,168 | **9.6%** |
| それ以外（地方） | 48,323 | 9,360 | **19.4%** |

**地方部ではOvertureの取りこぼし率が都市部の約2倍**。しかも地方は交通空白診断が主対象とする地域そのもの。「件数が多い」ことと「必要な場所をカバーしている」ことは別物だった。

これを全国分布で可視化したのが下図。左=Overture（青）、中=OSM（橙）、**右=OSMにしか無い店＝Overtureの穴（桃）**。右パネルの赤点が地方全域に散っており、地方偏在（19.4% vs 9.6%）を視覚的に裏付ける。

![Overture vs OSM 食品店POIの全国分布比較。左=Overture(n=170,899)、中=OSM(n=81,363)、右=OSMのみ＝Overtureの取りこぼし(n=12,528)](https://shiwaku.github.io/japan-mobility-ease-diagnosis/compare_overview_jp2.png)

> 図は定義準拠・名寄せ前（Overture 170,899件）の全国分布。名寄せ（後述）で件数は109,602件に減るが、この地方偏在の傾向は変わらない。**インタラクティブ版**（名寄せ済み）は [compare.html](https://shiwaku.github.io/japan-mobility-ease-diagnosis/compare.html) で触れる。

そして件数がOSMの倍ある理由を掘ると、**コンビニの重複**という沼にたどり着く。

## 4. 本題：コンビニが実数の2.2倍。その正体

調剤薬局を除いてもコンビニは **121,057件**。実店舗数は約5.6万店なので**約2.2倍**。特にセブンイレブンだけで48,129件ある（実数は約2.1万店）。

内訳を見ると、名前が**「セブン-イレブン」（ハイフン付、支店名なし）が25,737件**、「セブンイレブン」が897件——合わせて26,634件の汎用名エントリがあり、一方で**支店名つき「〇〇店」が20,076件**。後者は実店舗数とほぼ一致する。

つまり **支店名を持たない汎用名エントリが丸ごと重複しているのでは？** という仮説が立つ。これを空間検証で確かめた。

### 汎用名の正体を空間検証で確定する

「汎用名エントリが本当に実店舗の重複なら、近くに支店名つきの同チェーン店があるはず」——DuckDBで距離を測る。ここで**副産物のバグ**を踏んだ。

> **⚠️ DuckDB v1.3.2 の罠**: `ST_Distance_Spheroid` / `ST_DWithin_Spheroid` が座標に関わらず **`-nan` を返す**。近傍判定が無言で「0件ヒット」になり、危うく誤った結論を出すところだった。回避策は緯度補正つき等距円筒近似で自前計算：
>
> ```sql
> -- 2点間のメートル距離（緯度補正つき equirectangular 近似）
> sqrt(pow((lon2-lon1)*111320*cos(radians(lat1)),2) + pow((lat2-lat1)*110540,2))
> ```
>
> 総当たりを避けるため、`floor(lon/step), floor(lat/step)` でグリッドセルを作り8近傍セルだけ突合する。

判定半径を変えてセブンの汎用名（28,053件）を見ると：

| 判定半径 | 近くに支店名つきがある汎用名 | 割合 |
|---|---|---|
| 50m以内 | 8,564件 | 31% |
| 100m以内 | 9,311件 | 33% |
| 200m以内 | 10,817件 | 39% |
| **500m以内** | **24,868件** | **89%** |
| 660m格子内に皆無 | 1,354件 | 5% |

**89%が500m以内に支店名つきの相方を持つ**。汎用名は別店舗ではなく、**実店舗の重複レコードで座標が数百m単位でズレている**ものだと確定した。仮説は概ね正しかった。

### だが、単純な名寄せでは消せない

ここが面白いところ。「じゃあ近傍の重複を潰せばいい」——ところが**50m名寄せでは汎用名の18%しか除去できない**。座標ズレが数百mあるので、タイトな半径では捕まらないのだ。

かといって半径を広げると別の問題が出る。**都市部では本物の別店舗どうしが300m間隔で存在する**。500mで名寄せすると、別店舗まで誤って統合してしまう。実際、ブランド別に格子クラスタリングしても：

| チェーン | 生件数 | 格子500mでクラスタ化 | 実数 |
|---|---|---|---|
| セブンイレブン | 48,129 | 33,276（実数の1.5倍） | ~21,600 |
| ローソン | 29,242 | 23,966（実数の1.6倍） | ~14,600 |

**500m格子でも実数まで落ちない**。つまり重複レコードの座標ズレは500mを超えることが頻繁で、それは実店舗間隔より大きい。

> **原理的上限**: 座標誤差が実店舗間隔を超えるとき、純粋な空間名寄せは「重複除去」と「別店舗保持」を両立できない。

### 提供元を見れば消せる？ → それも無理だった

「重複は特定のダメな提供元が原因では？　そのソースを丸ごと落とせば？」——`sources` フィールド（提供元データセット）を含めて再抽出し、東京の小エリアで内訳を見た。

| 種別 | 主な提供元 | brand.wikidata |
|---|---|---|
| 支店名つき「〇〇店」 | ほぼ Foursquare | なし |
| 汎用名 | AllThePlaces / meta / Microsoft / Foursquare | AllThePlaces のみ有 |

想定と違った。**同じ1店舗が提供元ごとに別レコードとして残っている**（Overtureのconflation＝名寄せがコンビニで不完全）。しかも汎用名側の **AllThePlaces は公式店舗ロケーター由来で座標がむしろ正確**なことが多く、「汎用名＝低品質だから落とす」も成り立たない。

提供元別の件数を実数と突合すると、とどめを刺される：

| チェーン | Foursquare | meta | AllThePlaces | Microsoft | 実数 |
|---|---|---|---|---|---|
| セブン | 20,046 | 10,751 | 16,048 | 1,284 | ~21,600 |
| ローソン | 7,372 | 9,759 | 12,108 | 3 | ~14,600 |
| ファミマ | 6,834 | 6,775 | 9,034 | 551 | ~16,300 |

**どの単一提供元も実数に届かず（不完全）、かつ提供元間で完全一致しない**。union（全部足す）＝重複超過、単一提供元＝取りこぼし。真値は中間で、**Overture単独では正確な件数に到達できない**。

## 5. 現実解：単一提供元方式で名寄せ層を作る

「完璧は無理」と割り切り、**用途に合わせた**次善策を採った。

診断アプリの中核は「点到達性」（この住所から徒歩◯分に店があるか）。コンビニは高密度で最寄りは常に至近だから、数百mの座標ズレは徒歩閾値（500m/1km）の判定にほぼ響かない。しかも**主目的地であるスーパーは重複が軽微でクリーン**（支店名つき62%、実数≒2万店）。水増しが深刻なのはコンビニだけ。

そこで **5大コンビニチェーンだけ、チェーンごとに最網羅の単一提供元レコードを採用**する（各提供元は1店舗1レコードなので構造的に重複ゼロ）。その他カテゴリは重複軽微なので現状維持。

```sql
-- チェーン→最良提供元（実測の最網羅提供元）で1店舗1レコードに
best AS (SELECT * FROM (VALUES
  ('セブンイレブン','Foursquare'),
  ('ローソン','AllThePlaces'),
  ('ファミリーマート','AllThePlaces'),
  ('ミニストップ','meta'),
  ('デイリーヤマザキ','AllThePlaces')
) AS t(chain, provider))
-- 5大チェーンは datasets に最良提供元を含むレコードのみ、それ以外は全件維持
```

結果：

| チェーン | 採用提供元 | 名寄せ後 | 実数 | 網羅率 |
|---|---|---|---|---|
| セブンイレブン | Foursquare | 20,046 | ~21,600 | 93% |
| ローソン | AllThePlaces | 12,108 | ~14,600 | 83% |
| ファミリーマート | AllThePlaces | 9,034 | ~16,300 | 55% |
| ミニストップ | meta | 1,823 | ~1,900 | 96% |
| デイリーヤマザキ | AllThePlaces | 956 | ~1,400 | 68% |

**総件数 171,180 → 109,602件**（重複コンビニ約6.2万件を除去）。**コンビニ総計54,987件 ≒ 実数約56,000件**と整合した。構造的に重複ゼロ・座標良好。ファミマ・デイリーは最良提供元でも取りこぼしが残る（単一提供元の不完全性）が、重複ゼロと座標品質を優先して網羅率を割り切った層である。

## 6. カテゴリ別 最終比較：Overture vs OSM

名寄せ済みOvertureとOSMを、同じバケットに揃えて突合した（農水省定義準拠）。

| カテゴリ | Overture（名寄せ済） | OpenStreetMap | 差 |
|---|---:|---:|---:|
| スーパー | 18,521 | **20,348** | −1,827 |
| コンビニ | **54,987** | 48,676 | +6,311 |
| ドラッグストア | **7,735** | 4,554 | +3,181 |
| 食料品店（grocery） | 20,608 | 275 | +20,333 ⚠️ |
| 生鮮専門店・直売所 | 7,751 | 7,527 | +224 |
| **合計** | **109,602** | **81,380** | — |

読み取り：

- **カテゴリで優劣が入れ替わる**。スーパーはOSM有利、コンビニ・ドラッグストアはOverture有利。「どちらが網羅的か」はカテゴリ依存。
- **⚠️ 食料品店（grocery）の 20,608 vs 275 は比較不能**。これは網羅差ではなく**タグ体系の違い**。OSM日本では `shop=grocery` がほぼ使われず（個人商店は別タグ or 未マッピング）、Overtureの `grocery_store` が受け皿カテゴリになっているため。カテゴリ境界がソース間でズレる典型例で、この行だけで優劣は論じられない。
- 診断アプリの主目的地である**スーパーでOSMが上回る**点は、Overture単独運用時の取りこぼしリスクとして留意（マージ or OSM補完の検討材料）。

## 7. 教訓

1. **件数の多さ＝網羅性ではない**。重複・水増しを疑い、実数（公式店舗数など）と突合してから判断する。
2. **オープンPOIの重複はクロス提供元conflationの漏れ**として現れる。単一のダメソースを落とせば済む、という単純な話ではないことが多い。
3. **座標誤差が対象物の間隔を超えると、空間名寄せには原理的上限がある**。半径を上げれば別物を統合し、下げれば重複が残る。正確な件数が要るならグラウンドトゥルース（公式ロケーター／経済センサス小売業メッシュ）が要る。
4. **名寄せ方針は用途で決める**。点到達性の診断なら座標ズレの影響は小さく、統計・密度表示なら致命的。「何に使うか」を先に決めると割り切れる。
5. **GISの距離関数は動作検証してから使う**。`ST_Distance_Spheroid` が `-nan` を返すような環境依存バグは、無言で誤結論を生む。既知の2点で必ずサニティチェックする。

## 8. 再現手順（DuckDB）

全処理はDuckDB（httpfs + spatial 拡張）で再現可能。主要スクリプト：

| スクリプト | 役割 |
|---|---|
| `extract_overture_full.sql` | Overture S3から `sources`/`brand`/`country` 付きフルフィールドで日本の食料品POIを再抽出 |
| `dedup_convenience_spatial.sql` | コンビニ汎用名の重複を近傍判定で定量化（名寄せ検証） |
| `build_deduped_layer.sql` | 単一提供元方式で名寄せ層 `overture_food_deduped_jp.parquet` を生成 |
| `compare_sources_by_category.sql` | Overture（名寄せ済）vs OSM をカテゴリ別に件数突合 |
| `build_pmtiles.sh` | 名寄せ層から比較ビューワー用PMTilesを再生成（tippecanoe） |

抽出の骨子（Overture）：

```sql
INSTALL httpfs; LOAD httpfs; SET s3_region='us-west-2';
SELECT id, names.primary AS name, categories.primary AS category,
       brand.wikidata AS brand_wikidata,
       addresses[1].country AS country,
       list_transform(sources, s -> s.dataset) AS datasets,
       bbox.xmin AS lon, bbox.ymin AS lat   -- 点ジオメトリなのでWKBパース不要
FROM read_parquet('s3://overturemaps-us-west-2/release/2026-06-17.0/theme=places/type=place/*.parquet')
WHERE bbox.xmin BETWEEN 122.5 AND 154.0 AND bbox.ymin BETWEEN 24.0 AND 45.8
  AND categories.primary IN ('convenience_store','supermarket','grocery_store',
      'drugstore','butcher_shop','farmers_market','seafood_market', /* 除外用に */ 'pharmacy','discount_store','department_store');
```
