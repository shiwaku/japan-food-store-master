# 試作：施設マスター（after版 = geocode修正後データから）

> 対象: `facilities-all_after.csv`（geocodeQuery修正後・1,437,799許可レコード）
> 作成日: 2026-07-18
> 関連: `検証_食品営業許可データ_許可台帳としての重複構造.md` / PR #30（geocode修正）

## 目的
許可台帳（1行=1許可）を**物理施設単位（1行=1施設）**へ集約した施設マスターの試作。

## 方法
- **dedupキー**: `name + address`
- **代表座標**: 精度優先で1点を採用（元座標 > level 8 > 3 > 2 > 1）
- **集約列**: `license_count`（許可数）、`business_types`（業種をdistinct連結）、`license_nos`（許可番号連結）、phone/sources は代表行から
- 出力: `facility_master_after.csv`（344MB）/ `facility_master_after.parquet`（99MB, QGIS用・座標ありのみ）

## 結果
| 項目 | 値 |
|---|---:|
| 許可レコード（入力） | 1,437,799 |
| **施設マスター（name+address）** | **1,193,102** |
| GeoParquet 点数（座標あり） | 1,129,787 |

検証例: マックスバリュ東海（いなべ市北勢町阿下喜）→ 5許可を1施設に集約、
`business_types = 魚介類販売業; 食肉販売業; 飲食店営業; 菓子製造業`、代表 level 8。✅

## 発見・要改善（次の課題）

1. **★ business_type の表記が出典間で不統一** — 最大の課題。
   - 同じ「飲食店営業」が **`飲食店営業`（290,469）** と **`① 飲食店営業`（260,203）** の2表記で別物扱い。
     i2fas は丸数字プレフィックス付き、自治体データは無し。`飲食店営業（旧）` などの派生もある。
   - → `business_types` の連結が冗長化。**丸数字・空白・(旧)等を剥がして正規化**してから集約すべき。
2. **ノイズが残る**（要フィルタ）:
   - マスク名（`※※※`/`*`）: 20,451施設
   - 区域型/住所欠（`一円`/`全域`/`管内`/null）: 114,080施設
   - 座標なし: 63,315施設
   - 高license_count上位に**催事名**（「大北海道展」「美味コレクション」＝千駄ヶ谷五丁目＝百貨店催事）や
     テーマパーク（ナガシマ・鈴鹿）が混在。name=施設名でない行がある。
3. **name+address の限界**: 全角/半角・表記揺れで同一施設が分離し得る（住所正規化前のため）。
   name+座標キーとの併用や、住所正規化後のキーで精度向上の余地。

## 位置づけ
本試作は「許可台帳→施設マスター」の第一歩。実用には **(1)業種表記の正規化 → (2)ノイズ除去 →
(3)キーの頑健化（住所正規化/座標併用）** の順で磨く必要がある。買い物アクセス分析で使うなら
食料品店業態（届出のコンビニ/スーパー/その他食料飲料販売 等）で絞った上で施設集約する。
