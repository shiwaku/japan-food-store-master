## Q1. 食品営業許可・届出の対象業種は？

改正食品衛生法（令和3年6月施行）で3層構造。詳細は `整理_食品営業許可届出_対象業種.md`。

- **要許可32業種**: 飲食店営業(①)、菓子製造業、食肉処理業…製造業＋飲食店＋未包装の食肉/魚介販売 が中心。
- **要届出業種**: コンビニ、百貨店・総合スーパー、野菜果物・米穀・乳類・弁当販売、その他食料飲料販売業 など。
- **届出不要業種**: 輸入業、運搬・貯蔵業、常温長期保存包装食品販売 など。

> ⚠️ データの「営業の種類」列は**丸数字が2系統混在**（許可①〜㉜／届出①〜㉙で番号重複）。集計は業種名で寄せる。

## Q2. 許可32業種からスーパー・コンビニを抽出できる？

**できない（許可32には無い）。スーパー・コンビニは届出業種。**

- 抽出するなら届出コード（⑩コンビニ／⑪百貨店・総合スーパー／⑦野菜果物／⑬その他食料飲料販売 等）で絞る。
- ただし **悉皆データにはならない**。実測（i2fas全157ファイル・現役）:

| 業種 | i2fas件数 | 実数目安 | 充足 |
|---|---:|---:|---|
| コンビニエンスストア | 27,615 | 約56,000店 | **約半分** |
| 百貨店・総合スーパー | 28,187 | — | — |

- 弱点3つ: ①i2fasはオプトイン（賛同施設のみ）、②届出の登録は許可より薄い、③「その他の食料・飲料販売業」が受け皿肥大。
- **このデータの得意分野は飲食店営業（許可①）であって食料品小売店ではない**。→ 食料品店POIはOSM/Overtureが主、許可データはクロス材料。

## Q3. 「悉皆（しっかい）データ」とは？

対象を一件残らず全部網羅した完全な全数データ。サンプルや一部抽出ではない。
買い物難民診断のように「店の不在」を測る用途では、店の抜けが致命的なので悉皆性が要件になる。

## Q4. 全144万件は「日本全国で使える」データか？（悉皆性の観点）

**全国スコープではあるが、全国悉皆ではない。全国一律データとしては使えない。**

- **横（自治体カバレッジ）= 99.2%**（1,727/1,741市区町村）。47都道府県すべてに存在＝建付けは全国。
- **縦（各地の悉皆性）= 県ごとに桁違いで信用できない。** 件数が人口に全く比例しない:

| 県 | 人口(概算) | 件数 | 備考 |
|---|---:|---:|---|
| 長野県 | 約200万 | 65,439 | 県が全域OD（管轄主体型）＝厚い |
| 新潟県 | 約210万 | **2,730** | i2fas頼み＝ほぼ欠落（**同人口で長野の1/24**） |
| 神奈川県 | 約920万 | 28,412 | 川崎・藤沢・横須賀が404欠落 |
| 愛知県 | 約750万 | 21,638 | 名古屋市が404欠落で壊滅 |
| 東京都 | 約1400万 | 152,046 | 妥当 |

穴は2種類:
- **① ビルド固有**（本家ソース定義URLが恒常404の14件）: 愛知(名古屋)・神奈川(川崎/藤沢/横須賀)・青森・秋田・山形・福島・福井・松本・滋賀県・西宮・東大阪・四日市・横須賀。**URL更新で修正可能**。
- **② 構造的**（管轄主体がODを出さずi2fas頼み）: 新潟・茨城・宮崎・秋田 等。**供給側オプトインで修正困難**。

→ 件数の少なさが「店が無い」のか「データが無い」のか区別できない。網羅性を要する用途にはこのデータ単独では不十分。

## Q6. 施設検索API（Googleマップ的）の素材としての適性は？

**用途が変わると評価も変わる。** 検索APIは「存在志向（結果が返ればよい）」で悉皆不要 → Q4の否定評価は当たらない。
**素材としては筋が良い。ただしそのままGoogleマップ品質にはならず、3点が品質を縛る。**

### ① 座標精度（近接検索の生命線）← 最重要
| 精度 | 件数 | 近接検索 |
|---|---:|---|
| 元座標（source由来） | 828,882 | ◎ |
| level 8 街区・地番 | 207,934 | ◎ |
| level 3 丁目重心 | 208,348 | ○（数百m内） |
| level 2 市区町村点 | 28,889 | △ |
| **level 1 県代表点** | **82,184** | **✗ 1点に集積し検索を壊す** |

→ level 1の8万件は県代表点1箇所に集積。「現在地周辺」検索でその点付近に8万件がドバッと出る／他所では出ない＝**除外/フラグ必須**。

#### ★ level 1（82,184件）の根本原因＝主因はpipeline（ツールのバグでも非対応でもない）
実データ調査で判明。level 1住所に空欄は0件（`city`判明70,453件）。4分類:

| 分類 | 件数 | 割合 | 誰の問題か |
|---|---:|---:|---|
| **C 市名判明・通常住所** | **61,078** | **74%** | **pipeline**（完全な住所を組み立てず渡している。修正可能） |
| A 区域型（一円/全域/管内） | 19,402 | 24% | 誰の問題でもない（行商/自販機/区域許可＝点で表せない。正しく県レベル） |
| D 市名不明・住所も粗い | 1,701 | 2% | 元データ品質 |
| B 京都通り名（○○通△△上ル） | 3 | ~0% | ツール非対応（だが極少） |

- **ツールのバグではない**: 市名なし「兵庫県東灘区…」に県代表点を返すのは仕様どおり（曖昧入力に無理な推測をしない）。
- **ツール非対応でもない**: 完全な住所「兵庫県神戸市東灘区…」なら level 8 を返す。非対応は京都通り名のみ＝3件。
- **主因(74%)はpipelineの欠陥**: `geocodeQuery`（crawl.js:783）が **都道府県しか前置せず、既に `city` 列で判明している市区町村名を使わない**。
  神戸市・仙台市・富山県等は住所列が区/町名始まり（例「東灘区魚崎北町1丁目6‐7」、市名は別列）→ 渡す文字列が
  「兵庫県東灘区…」＝**神戸市欠落** → ジオコーダーが市名なしの区を解決できず県代表点へ転落。**持っている市名を捨てている**。
- **実証**（normalize-japanese-addresses、同一住所比較）:
  - `兵庫県東灘区魚崎北町1丁目6‐7` → **level 1**（34.691,135.183＝県代表点）
  - `兵庫県神戸市東灘区魚崎北町1丁目6‐7` → **level 8**（34.718,135.275＝街区精度、**約3km改善**）
- 修正はcrawl.js側で市名も前置（住所が既に区名始まりの場合の二重化に注意）。約6.1万件が level 1 → level 8/3 に改善見込み。**本家へ報告・修正の価値大**。

#### ★ 修正の実測検証（サンプル）
`geocodeQuery` に市名前置＋区名二重化ガードを実装し、level 1 のC分類レコードで before/after を実測:

| | level分布 | 近接検索可(lv3/8) |
|---|---|---|
| 現行(pref only) | すべて **level 1** | 0% |
| 修正案(pref+city) | level 8×29 / 3×18 / 2×2 | **95.9%** |

→ **level 1 が完全に解消**し、約96%が丁目〜街区精度に回復（うち level 8 街区精度が約6割）。

#### ★ before/after 成果物
本家 `japan-facilities-address` の `crawl.js:geocodeQuery` を修正し、pipeline全体を再実行して
`facilities-all.csv` の**修正前・修正後**を生成（クローン先に保存）:
- `facilities-all_before.csv` … 修正前（都道府県のみ前置）
- `facilities-all_after.csv` … 修正後（市区町村も前置）

同一フォーマット・同一行順（総行数1,437,799で不変）で、1件ずつ before/after を突合できる。差分は主に神戸市・仙台市・富山県等の
`lat/lng/geocoding_level`（level 1 → 8/3）。**issue #11（naogify、全国網羅追加計画）とは別の未報告バグ**なので、
この before/after を実証データとして PR に添付できる。

**全スケール実測（before → after）**:

| geocoding_level | before | after | 差 |
|---|---:|---:|---:|
| 1（県代表点） | 82,184 | **18,405** | **−63,779（−77.6%）** |
| 2（市区町村点） | 28,889 | 36,488 | +7,599 |
| 3（丁目重心） | 208,348 | 230,384 | +22,036 |
| 8（街区・地番） | 207,934 | **242,079** | +34,145 |

→ level 1 の 63,779 件が解消（うち **約56,181 件が丁目〜街区精度に回復**、7,599 件が市区町村点へ）。
残る level 1 = 18,405 件は「県下一円/行商/自販機」等の本来点で表せないデータ。総行数・座標あり数は不変＝**データ欠損ゼロ、精度のみ改善**。

### ② 地域カバレッジ穴 → 特定地域で結果ゼロ
名古屋・新潟・神奈川がほぼ空 → その地域は検索結果が返らない（製品欠陥）。①修正可能な死にURL14件の穴が特に痛い。

### ③ 鮮度・廃業・重複
- i2fasの `_all` は**廃業施設を含む**（要除外。merged側の扱い要確認）。
- 業種違いの重複は本家が名前+座標でdedup（search-index=576,202ユニーク）。
- スナップショット時点混在（京都 R3.3末等）。

### 結論（検索API用途）
| 観点 | 評価 |
|---|---|
| 用途の妥当性 | **妥当**（悉皆不要。57.6万ユニークは検索母数として十分） |
| そのままGoogleマップ級か | **ならない**（①〜③が縛る） |
| 最優先の改善 | **level 1（8万件）の除外/フラグ** → 死にURL14件更新 → 廃業除外 |

## Q7. 現在のジオコーディングは何のツール？座標精度は？

**`@geolonia/normalize-japanese-addresses` v3.1.3（Geolonia製）。**

- `crawl.js` が `normalize(住所)` を呼び、返り値 `point.lat/lng` と `point.level` を採用（8並列、`.cache/geocode-cache.json`にキャッシュ）。
- 実体は**「住所正規化＋到達レベルの代表点」を返すツール**。返す精度の上限は **level 8＝街区・地番**（建物入口・屋根精度ではない）。
- 住所が粗い/パース不可だと level 3（丁目）→2（市区町村）→1（県）に段階劣化。level 1の8万件＝「県までしか解決できなかった」住所。
- 座標精度の上限＝このツールの仕様（街区止まり）＋入力住所の質で決まる。

**検索API品質を上げる選択肢**:
- (a) `geocoding_level` で足切り／フラグ（例: level≧3 のみ採用、level 1は除外）
- (b) 入力住所のクレンジング（i2fas等の粗い住所を整える）
- (c) より高精度なジオコーダー（Geolonia API の建物レベルデータ等）への差し替え

## 総括

- **買い物難民分析（悉皆が要件）**: このデータ単独では**不可**。県ごとの網羅度差が致命的。POI（OSM/Overture）主で。
- **施設検索API（存在志向・悉皆不要）**: **素材として妥当**。ただし座標精度フィルタ（level 1除外）＋死にURL補修＋廃業除外が前提。
- **共通の弱点**: ①ビルド固有の死にURL14件（修正可能）②i2fas依存の構造的穴（修正困難）③座標精度のフォールバック混在。
  - ③の主因は**ジオコーダーの限界ではなくpipelineバグ**（`geocodeQuery`が市区町村を前置しない）。level 1 の約6万件は正確な住所を持つのに県代表点へ転落しており、**市名前置の修正で street/block 精度へ回復可能**。元データの品質問題ではない。
