# 検証：食品営業許可データの重複構造（施設マスタではなく「許可台帳」）

> 対象: 大橋本家 `japan-facilities-address` の `facilities-all.csv`（1,437,799行、i2fas含む）
> 作成日: 2026-07-18
> 関連: `検証_食品営業許可届出データ_網羅性と検索API適性.md` / `整理_食品営業許可届出_対象業種.md`

## 結論

**このデータは「施設マスタ」ではなく「許可台帳」。1行 = 1許可（ライセンス）であって、1施設 = 1行ではない。**
同一の物理施設が複数行になる。物理店舗数や買い物アクセスを扱うなら、施設単位へ集約（dedup）が必要。

## 2. 実例

マックスバリュ東海（いなべ市北勢町阿下喜3325-1、座標同一）が **4行**:

| business_type | license_no |
|---|---|
| 飲食店営業 | 桑2501-0798 |
| 飲食店営業 | 桑2501-0797 |（← 同一業種で2件＝②）
| 食肉販売業 | 桑2503-0011 |
| 魚介類販売業 | 桑2504-0013 |

→ 全列一致dedupは `business_type`/`license_no` が異なるため消さない。本家READMEも
「業種違いで別レコードになっている同一施設は、許可レコードとして残す」と明記＝**意図的**。

## 3. 用途別の扱い

| 用途 | 重複の扱い |
|---|---|
| **店舗数カウント／買い物アクセス分析** | **物理店舗へdedup必須**（name+address or name+座標）。でないと1店を複数回カウント |
| 施設検索API | 本家は search-index で name+座標 dedup 済み（※コミット版README 576,202は i2fas無しビルドの値。今回のi2fasありでは name+座標 約1,109,146） |
| 許可台帳としての参照 | そのまま（1許可1行が正しい） |

## 4. 施設マスタ化の注意点（別途作業）

物理施設マスタを作る場合の論点:
- **dedupキー**: (A) name+座標（同一地点の同名を1点に。表示・検索向き） / (B) name+address（住所単位）。
- **業種の保持**: 畳む際に `business_type` をリスト連結（例「飲食店営業; 食肉販売業; 魚介類販売業」）して情報を残す。
- **ノイズ**: 重複上位に `※※※※※`（マスク済み個人情報）、`株式会社フジ`（address=None＝本社名のみ）、`県内一円`（区域型）が混在。単純dedupで誤集約しないようフィルタが要る。
- **座標のばらつき**: 同一施設でも行により座標が微妙に違う場合がある（元座標 vs ジオコーディング混在）。name+座標dedupは座標一致が条件なので、座標が割れると畳まれ残る。

→ 施設マスタは本データからの派生成果物として**別途構築**する。まずは許可台帳としてそのまま扱う。
